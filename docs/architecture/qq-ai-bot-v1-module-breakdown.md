# 模块拆分与验收指导文档规划（v1 实现分解母文档）

## 摘要

- 新增一份“后续实现规划母文档”，用于指导 v1 如何按模块拆分、按 checkpoint 顺序推进、以及每个模块应如何验收。
- 文档只覆盖 v1 当前要实现的模块，不展开未来预留模块；但会在文档开头重申已有架构约束，确保拆分不破坏后续扩展位。
- 拆分粒度采用中等粒度：每个 checkpoint 应能单独做一次后续 plan 和一轮实现，通常对应一个明确子目标和一套验收标准。
- 验收策略采用平衡模式：能低成本独立测试的模块必须有单模块验收；明显依赖外部协议端、第三方网络或完整链路的部分，可降级为集成验收，但必须写明为何不做独立验收、以及集成验收的最小覆盖范围。

## 文档定位与输出结构

- 建议文档路径：`docs/architecture/qq-ai-bot-v1-module-breakdown.md`
- 文档目标：
  - 明确 v1 的模块边界
  - 说明每个模块的职责、输入输出、依赖、实现前置条件
  - 给出每个模块的验收方式：独立验收、集成验收、人工验收补充
  - 定义分步骤实现顺序和 checkpoint 清单，供后续逐个模块继续做 plan
- 文档固定结构：
  - 标题与目的
  - 约束前提
  - 模块拆分清单
  - 模块验收标准
  - 集成验收边界说明
  - checkpoint 顺序清单
  - 风险与后续计划接口

## 模块划分方案

- `M0 项目骨架与运行时约束`
  - 职责：建立目录结构、配置加载入口、依赖注入主线、统一启动方式、基础日志和错误边界。
  - 输入输出：输入为配置文件与环境变量；输出为可启动的空壳服务与模块注册入口。
  - 依赖：无业务前置，是所有模块基础。
  - 验收：应可独立验收。
- `M1 标准事件模型与 Adapter 契约`
  - 职责：定义 `NormalizedEvent`、回复接口、消息场景抽象、统一 adapter 契约。
  - 输入输出：输入为原始消息事件映射需求；输出为稳定的内部事件接口与 adapter side contract。
  - 依赖：依赖 M0。
  - 验收：应可独立验收。
- `M2 QQ Personal Adapter（OneBot/NapCat 接入层）`
  - 职责：接收 OneBot v11 事件、转换为标准事件、回发消息、屏蔽协议细节。
  - 输入输出：输入为 OneBot 事件；输出为内部事件与统一发送能力。
  - 依赖：依赖 M0、M1。
  - 验收：以模块内契约测试 + 集成验收为主。
- `M3 Router 与插件执行主线`
  - 职责：把标准事件分流到命令、帮助、原神查询、闲聊处理链；提供统一插件调用入口。
  - 输入输出：输入为 `NormalizedEvent`；输出为 `PluginResult` 或回复动作。
  - 依赖：依赖 M1。
  - 验收：应可独立验收。
- `M4 Session、频控与权限控制`
  - 职责：会话上下文、消息去重、基本限流、用户级功能开关和敏感能力保护。
  - 输入输出：输入为事件与用户标识；输出为会话状态、去重结果、放行或拒绝决策。
  - 依赖：依赖 M0、M1，可被 M3 调用。
  - 验收：大部分应可独立验收。
- `M5 Storage 抽象与 SQLite/Redis 实现`
  - 职责：定义存储接口，并实现配置、绑定关系、提醒任务、缓存、去重存储。
  - 输入输出：输入为领域对象；输出为持久化结果、查询结果、缓存状态。
  - 依赖：依赖 M0。
  - 验收：应可独立验收。
- `M6 Common 插件包`
  - 职责：帮助、健康检查、基础设置、基础回显等通用能力。
  - 输入输出：输入为标准事件与上下文；输出为确定性回复。
  - 依赖：依赖 M3，必要时依赖 M4、M5。
  - 验收：应可独立验收。
- `M7 原神静态资料 Provider 与查询插件`
  - 职责：加载本地静态资料，提供角色、武器、材料等静态查询。
  - 输入输出：输入为查询词；输出为标准化静态资料结果。
  - 依赖：依赖 M3，可选依赖 M5。
  - 验收：应可独立验收。
- `M8 Enka Provider 与公开面板查询插件`
  - 职责：封装 Enka.Network 访问、TTL 缓存、公开 UID 面板摘要查询。
  - 输入输出：输入为 UID；输出为公开角色面板摘要。
  - 依赖：依赖 M5、M3。
  - 验收：provider 层可独立测试一部分，但完整功能以集成验收为主。
- `M9 Hoyo/genshin.py Provider 与账号绑定能力`
  - 职责：cookie 绑定、实时便笺、签到、战绩摘要。
  - 输入输出：输入为用户凭据与功能请求；输出为账号类查询或操作结果。
  - 依赖：依赖 M5、M3、M4。
  - 验收：核心状态流转可独立测试，第三方调用结果以集成验收为主。
- `M10 LLM Provider 与基础闲聊插件`
  - 职责：抽象模型 provider、接入默认模型、实现基础闲聊和简单规则升级。
  - 输入输出：输入为消息上下文；输出为模型回复与 usage、cost 记录。
  - 依赖：依赖 M3、M5，可选依赖 M4。
  - 验收：provider 封装与路由规则可独立验收，真实模型调用需集成验收补足。
- `M11 Scheduler 与提醒任务`
  - 职责：定时任务注册、恢复、提醒发送、失败重试的最小闭环。
  - 输入输出：输入为提醒配置；输出为按时触发的消息发送。
  - 依赖：依赖 M2、M5、M6 或 M9。
  - 验收：调度逻辑可独立验收，实际“到点发 QQ 消息”需集成验收。
- `M12 Observability 与运行诊断`
  - 职责：结构化日志、错误分类、基础 metrics 占位、成本记录、健康检查端点。
  - 输入输出：输入为运行时事件；输出为日志、诊断状态、健康检查结果。
  - 依赖：横切全局，依赖 M0。
  - 验收：大部分可独立验收。
- `M13 v1 端到端集成收口`
  - 职责：把前述模块串联成可运行系统，覆盖最小业务闭环。
  - 输入输出：输入为真实 QQ 消息；输出为可观察、可回复、可查询、可提醒的完整系统。
  - 依赖：依赖所有 v1 必需模块。
  - 验收：只做集成验收，不再追求模块级隔离。

## 模块验收标准设计

- 独立验收模块必须具备：
  - 明确的输入输出接口，不依赖 QQ 原始对象
  - 可通过 fake、mock provider 或内存实现替代真实外部依赖
  - 至少一组成功路径和一组失败路径验收标准
  - 明确“完成即稳定”的最小行为，不把开放性留给实现者二次决定
- 推荐独立验收的模块：
  - `M0`、`M1`、`M3`、`M4`、`M5`、`M6`、`M7`、`M12`
- 推荐“独立 + 集成混合验收”的模块：
  - `M2`、`M8`、`M9`、`M10`、`M11`
- 明确只做集成收口的模块：
  - `M13`
- 需要在文档中为每个模块写明以下字段：
  - 模块目标
  - 非目标
  - 依赖前置
  - 可独立验收的边界
  - 验收方式
  - 最小通过标准
  - 是否需要人工冒烟
  - 是否需要人工配合
  - 人工验证前置条件
  - 人工验证最小场景
  - 人工验证失败观察点
  - 后续扩展接口影响

## 人工冒烟与协作测试要求

- 当模块的核心价值依赖真实外部系统联通，而自动化无法低成本覆盖真实风险时，必须补充人工冒烟测试方案。
- 人工冒烟不替代自动化：
  - 内部逻辑、状态流转、缓存行为、路由规则仍应优先自动化验证
  - 人工冒烟只覆盖“真实依赖接通”和“关键链路最终可用”
- 对需要人工冒烟的模块，后续子计划至少要写清：
  - 为什么不能只靠自动化验收
  - 最小人工冒烟场景
  - 冒烟前置条件
  - 通过标准
  - 失败观察点
  - 无法自动化覆盖的风险
- 仅对高风险模块单独区分“是否需要人工配合”：
  - 需要第二个 QQ 账号、测试群、真实 cookies、真实业务账号、真实模型 key、独立观察者或时间窗口时，应标记为可能需要人工配合
  - 文档中要写清协作方角色、协作方提供物、无法由单人本地模拟替代的步骤
- 高风险模块的人工测试要求：
  - `M2 QQ Personal Adapter`
    - 必须补人工冒烟
    - 建议标记为“可能需要人工配合”
  - `M8 Enka Provider`
    - 必须补人工冒烟
    - 通常不需要人工配合
  - `M9 Hoyo/genshin.py Provider`
    - 必须补人工冒烟
    - 建议标记为“需要人工配合或真实业务数据配合”
  - `M10 LLM Provider`
    - 必须补人工冒烟
    - 通常不需要第二人配合，但需要真实 key 和真实模型配额
  - `M11 Scheduler`
    - 必须补人工冒烟
    - 高概率需要人工配合或至少人工值守
  - `M13 v1 端到端集成收口`
    - 必须补完整人工冒烟
    - 对部分场景要允许人工配合

## 不适合单独验收的情况与说明规则

- 当模块主要价值来自“外部真实系统联通”而不是内部逻辑时，可降低独立验收优先级。
- 允许只做集成或以集成为主的典型场景：
  - OneBot、NapCat 真连接与回发链路
  - Enka.Network 真实返回兼容性
  - genshin.py 对米游社 cookies 的真实联通
  - 调度任务触发后的真实 QQ 发消息链路
  - 真实模型 provider 调用、usage 和记费闭环
- 文档中必须写清：
  - 为什么单独验收收益低
  - 哪部分仍需单测或契约测
  - 哪个集成场景能覆盖该模块最关键风险
  - 哪些失败模式必须在集成阶段显式观察
  - 是否还需要人工冒烟补位
  - 是否需要人工配合才能完成最终确认

## Checkpoint 实现顺序

- `CP1 基础骨架`
  - 覆盖 `M0`
  - 目标：项目可启动、配置可加载、日志与错误边界存在
  - 这是后续所有模块的共同前置
- `CP2 统一事件与执行主线`
  - 覆盖 `M1 + M3`
  - 目标：内部事件模型、router、插件契约跑通，哪怕暂时没有真实 QQ 接入
  - 产出：可用 fake event 驱动 help、echo 这类最小插件
- `CP3 存储、会话、频控基础能力`
  - 覆盖 `M4 + M5`
  - 目标：形成稳定的状态管理和存储抽象，避免后续插件绕过边界直接落库
- `CP4 QQ 接入闭环`
  - 覆盖 `M2 + M6`
  - 目标：OneBot 事件进入系统后，至少能跑通 help、echo、health 这类通用回复
  - 这是第一个真实链路闭环 checkpoint
- `CP5 原神静态资料能力`
  - 覆盖 `M7`
  - 目标：不依赖外网即可完成第一批原神查询
  - 这是第一个稳定、低风险、可频繁验证的业务模块
- `CP6 Enka 公开面板能力`
  - 覆盖 `M8`
  - 目标：公开 UID 查询 + TTL 缓存跑通
  - 验证外部 provider 接入模式是否成立
- `CP7 米游社账号能力`
  - 覆盖 `M9`
  - 目标：cookie 绑定、便笺、签到、战绩摘要的最小闭环
  - 这是第一批高风险外部依赖模块，必须单独作为 checkpoint
- `CP8 模型闲聊能力`
  - 覆盖 `M10`
  - 目标：普通闲聊和简单模型路由跑通，记录 usage、cost
  - 不引入复杂 tool orchestration
- `CP9 提醒与调度`
  - 覆盖 `M11`
  - 目标：用户设置提醒后，系统能够恢复并按时发送
  - 这是第一批跨模块组合能力
- `CP10 观测与收口`
  - 覆盖 `M12 + M13`
  - 目标：完善日志、健康诊断、端到端回归清单，形成 v1 可持续迭代基线

## 每个 Checkpoint 在文档中的记录方式

- 对每个 checkpoint 固定写明：
  - checkpoint 目标
  - 涉及模块
  - 前置 checkpoint
  - 本 checkpoint 完成后应新增的可见能力
  - 本 checkpoint 必过验收项
  - 本 checkpoint 是否包含人工冒烟项
  - 若包含，需写清由谁执行、需要什么外部条件、通过标准是什么
  - 暂不处理的问题
- 文档中不写具体代码任务拆分到文件级，不规定实现细节，只定义“下一轮 plan 必须回答的问题边界”。

## 测试与验收组织方式

- 自动化优先但不强求极致隔离，采用四层验证：
  - `模块级`：纯内存、fake provider、契约测试，覆盖路由、状态、存储、静态查询等稳定逻辑
  - `适配或 Provider 级`：对外部依赖做映射测试、错误处理测试、缓存行为测试
  - `系统级`：真实 OneBot、真实外部 provider、真实定时触发、真实模型调用闭环
  - `人工冒烟或协作级`：覆盖真实依赖接通、最终用户视角可用性、无法低成本自动化的关键风险
- 文档需要明确每类测试对应的目标：
  - 模块级验证“边界没坏”
  - provider 级验证“第三方适配没错位”
  - 系统级验证“真实链路可用”
  - 人工冒烟级验证“最终链路真的能被人使用并观察到结果”

## 假设与默认值

- 本文档只服务于 v1 当前实现计划，不展开未来 `ToolRegistry`、`AgentOrchestrator`、`QQOfficialAdapter` 的实施拆分。
- 后续每次子计划都以某个 checkpoint 或单个模块为范围，不跨越多个高风险模块同时推进。
- 默认允许部分外部依赖模块以集成验收为主，但必须在文档中写出原因，而不是模糊地“后续再测”。
- 默认实现顺序优先保证“内部边界先稳定，再接外部系统”，避免一开始就被 NapCat、Enka、米游社、模型接口牵着走。
